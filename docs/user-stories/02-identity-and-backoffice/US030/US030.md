# US030 - Authentication and Authorization

## 1. Requirements Engineering

### 1.1. User Story Description

As a Project Manager, I want the system to support and apply authentication and authorization for all its users and functionalities.

This user story ensures that only authenticated users can access the system and that each user can only execute the functionalities allowed by their role and current access status.

---

### 1.2. Customer Specifications and Clarifications

**From the specifications document:**

* A user is someone with access to the system.
* A user is identified by a unique valid email from the list of valid email domains.
* A user also has a name and phone number.
* Users must authenticate into the system to do anything.
* An AlSafe user needs to have an active security clearance that automatically expires at a given date.
* Users need to have periodic skills assessment, according to regulations, every 5 years.
* The Admin or the Backoffice Operator can update the security clearance and skills assessment information.
* Currently, management has not yet decided if a user can have multiple roles in the system.
* Right now, no user has multiple roles, but that may happen in the future.
* The number of roles may also increase in the future.
* Authentication and authorization are required by the system's non-functional requirements.

**From the client clarifications:**

No additional client clarifications are currently available.

---

### 1.3. Acceptance Criteria

* **AC1:** The system must require authentication before allowing access to any protected functionality.
* **AC2:** A user must authenticate using valid credentials.
* **AC3:** A disabled user must not be allowed to authenticate.
* **AC4:** A user with an expired security clearance must not be allowed to access protected functionalities.
* **AC5:** A user with an expired skills assessment must not be allowed to access protected functionalities.
* **AC6:** The system must verify if the authenticated user has the required role or permission before executing a functionality.
* **AC7:** If authentication fails, the system must display an appropriate error message.
* **AC8:** If authorization fails, the system must display an appropriate access denied message.
* **AC9:** The authorization model should be prepared to support multiple roles per user in the future, even if users currently have only one role.
* **AC10:** Authentication and authorization rules must be applied consistently across all system functionalities.

---

### 1.4. Found out Dependencies

* This user story depends on the existence of a User concept in the domain model.
* This user story is related to US031, because users must be registered before they can authenticate.
* This user story is related to US032, because disabled users must not be allowed to authenticate.
* This user story is related to US033, because user status information is relevant for administration.
* This user story affects all functional user stories, since access control must be enforced globally.

---

### 1.5. Input and Output Data

**Input Data:**

* Typed data:
    * Email
    * Password

**Output Data:**

* In case of successful authentication:
    * Authenticated user information
    * User role or roles
    * Access confirmation

* In case of failed authentication:
    * Error message explaining that authentication failed

* In case of failed authorization:
    * Error message explaining that the user does not have permission to execute the requested functionality

### 1.7. Other Relevant Remarks

* This user story is transversal to the entire system.
* The system should not allow anonymous access to protected functionalities.
* The authorization mechanism should avoid hardcoding permissions directly inside user interface classes.
* Since the number of roles may increase in the future, the design should avoid assuming a fixed and closed set of roles.
* Although users currently do not have multiple roles, the model should be flexible enough to support that possibility later.

## 2. Analysis

### 2.1. Relevant Domain Concepts

* **User:** person with access to the system.
* **Email:** unique identifier used for authentication.
* **Password/Credential:** secret information used to prove user identity.
* **Role:** defines the type of access a user has.
* **Permission:** represents an allowed action in the system.
* **Security Clearance:** authorization condition that expires at a given date.
* **Skills Assessment:** periodic validation that must be renewed every 5 years.
* **User Status:** indicates whether the user is enabled or disabled.

---

### 2.2. Business Rules

* A user must authenticate before using any protected functionality.
* A user must have a unique valid email.
* A disabled user cannot authenticate.
* A user with expired security clearance cannot access protected functionalities.
* A user with expired skills assessment cannot access protected functionalities.
* A user can only execute a functionality if their role or permissions allow it.
* The system should be prepared to support multiple roles per user in the future.
* Authentication failure and authorization failure must be treated as different cases.

---

### 2.3. Preconditions

* The user must already be registered in the system.
* The user must provide authentication credentials.
* The requested functionality must declare the required permission or role.

---

### 2.4. Postconditions

**Successful authentication:**

* The user is considered authenticated.
* The system allows the user to request protected functionalities.

**Successful authorization:**

* The requested functionality is executed.

**Failed authentication:**

* The user remains unauthenticated.
* No protected functionality is executed.

**Failed authorization:**

* The user remains authenticated.
* The requested functionality is not executed.

---

## 3. Design

### 3.1. System Sequence Diagram

The SSD represents the interaction between a generic user and the system when trying to authenticate and access a protected functionality.

The user first submits credentials. The system validates those credentials, verifies the user's status, security clearance and skills assessment. If authentication succeeds, the user may request a protected functionality. The system then checks if the authenticated user has the required authorization.

---

### 3.2. Suggested Design Elements

Possible classes/interfaces involved:

* `User`

* `Password`
* `Role`
* `Permission`
* `SecurityClearance`
* `SkillsAssessment`
* `AuthenticationService`
* `AuthorizationService`
* `UserRepository`
* `AuthenticatedUserSession`

---

### 3.3. Applied Patterns

* **Controller:** receives authentication and access requests.
* **Service:** coordinates authentication and authorization logic.
* **Repository:** retrieves user data.
* **Entity:** represents users and their access state.
* **Value Object:** represents email, role, password, clearance expiration date and skills assessment date.
* **Guard/Policy:** centralizes authorization rules.

---

### 3.4. Tests to be implemented

* Verify that a valid active user can authenticate.
* Verify that an unknown email cannot authenticate.
* Verify that an incorrect password fails authentication.
* Verify that a disabled user cannot authenticate.
* Verify that a user with expired security clearance cannot access protected functionalities.
* Verify that a user with expired skills assessment cannot access protected functionalities.
* Verify that an authenticated user with the correct role can execute a protected functionality.
* Verify that an authenticated user without the required role cannot execute a protected functionality.
* Verify that the authorization model can handle more than one role per user.
